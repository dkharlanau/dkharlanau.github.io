# AGENT_TASK_PATTERNS.md

Safe task patterns for AI agents working in `dkharlanau.github.io`.
Last updated: 2026-06-09

## Purpose

This document gives future AI agents concrete, safe task templates. Each pattern includes what to do, what files to touch, what to avoid, how to validate, and what to report.

---

## Pattern 1: Adding a New Atlas Diagnostic Page

### Goal
Create a new diagnostic page in the Atlas that follows the established structure and starts at Level 1 (unverified).

### Allowed Files
- `atlas/diagnostics/<slug>.md` — the new page
- `atlas/diagnostics/index.md` — update the section index to list the new page
- `atlas/manifest.json` — will be regenerated by script

### Forbidden Files / Actions
- Do not edit `llms-full.txt` directly
- Do not set `verified: true`
- Do not set `robots: index,follow`
- Do not set `sitemap: true`
- Do not expose client names, ticket numbers, or proprietary details
- Do not invent external links or citations

### Required Frontmatter
```yaml
---
layout: default
title: "<Descriptive Title>"
description: "<Conservative description>"
permalink: /atlas/diagnostics/<slug>/
atlas_section: diagnostics
domain: SAP AMS
subdomain: <e.g., Master data and MDG>
concept_type: diagnostic guide
sap_area: "<SAP area>"
business_process: "<Business process>"
status: needs_verification
verified: false
last_reviewed: <YYYY-MM-DD>
author: Dzmitryi Kharlanau
tags:
  - <tag1>
  - <tag2>
related:
  - /atlas/diagnostics/<existing-related-page>/
robots: noindex,follow
sitemap: false
---
```

### Validation Commands
```sh
bundle exec jekyll build
python3 scripts/check_links.py _site
python3 scripts/check_seo.py _site
```

### Final Report Requirements
- Confirm the new page builds without errors
- Confirm the page is reachable at the expected URL
- Confirm frontmatter includes `status: needs_verification`, `verified: false`, `robots: noindex,follow`, `sitemap: false`
- List related pages that were cross-linked
- Note any gaps or TODOs for human review

## Pattern 2: Adding a New Scenario Page

### Goal
Create a new business-pain-to-diagnostic-workflow scenario page.

### Allowed Files
- `scenarios/<slug>.md` — the new page
- `scenarios/index.md` — update the scenario index

### Forbidden Files / Actions
- Do not set `verified: true`
- Do not set `robots: index,follow`
- Do not set `sitemap: true`
- Do not expose client names or proprietary details
- Do not invent Atlas pages that do not exist

### Required Frontmatter
```yaml
---
layout: default
title: "<Business Pain Scenario>"
description: "<Conservative description>"
permalink: /scenarios/<slug>/
last_modified_at: <YYYY-MM-DD>
status: needs_verification
verified: false
author: Dzmitryi Kharlanau
robots: noindex,follow
sitemap: false
---
```

### Content Structure
Follow the consistent scenario pattern:
1. Business pain
2. Process context
3. SAP touchpoints
4. Root causes (master data / configuration / integration)
5. Cost drivers
6. Diagnostic workflow
7. Solution patterns
8. AI / automation opportunity (conservative boundaries)
9. Related Atlas links

### Validation Commands
```sh
bundle exec jekyll build
python3 scripts/check_links.py _site
```

### Final Report Requirements
- Confirm the new scenario builds without errors
- Confirm all related Atlas links exist and are reachable
- Confirm frontmatter is Level 1
- Note any missing Atlas deep-links that should be created later

## Pattern 3: Verifying and Promoting a Draft Page

### Goal
Move a page from Level 1 to Level 2 after human review.

### Allowed Files
- The specific page file being promoted

### Forbidden Files / Actions
- Agents may **not** perform this task autonomously
- This pattern is for human-in-the-loop workflows only
- An agent may prepare a promotion checklist but must not change `verified: false` to `verified: true`

### Human Review Checklist
1. [ ] Factual accuracy verified against SAP documentation or known system behavior
2. [ ] No client names, ticket numbers, or proprietary details exposed
3. [ ] No unsupported claims or speculative language
4. [ ] Related links are accurate and not broken
5. [ ] Frontmatter updated: `status: reviewed`, `verified: true`, `robots: index,follow`, `sitemap: true`
6. [ ] `last_reviewed` date set to today
7. [ ] `last_modified_at` updated if content changed

### Post-Promotion Commands
```sh
python3 scripts/generate_atlas_artifacts.py
bundle exec jekyll build
python3 scripts/check_links.py _site
python3 scripts/check_seo.py _site
```

### Final Report Requirements
- List the promoted page(s)
- Confirm `llms-full.txt` was regenerated
- Confirm sitemap artifacts were updated
- Note any remaining gaps

## Pattern 5: Updating LLMs / Sitemap Artifacts

### Goal
Regenerate machine-readable discovery files after content changes.

### Allowed Files
- Generated files: `llms-full.txt`, `atlas/manifest.json`, `ai/catalog.json`
- These are regenerated by scripts, not edited directly

### Forbidden Files / Actions
- Do not manually edit `llms-full.txt`
- Do not manually edit `sitemap-*.xml`
- Do not include Level 1 pages in generated artifacts

### Commands
```sh
python3 scripts/generate_atlas_artifacts.py
python3 bin/enrich_datasets.py
python3 bin/generate_dataset_pages.py
bundle exec jekyll build
```

### Final Report Requirements
- Confirm `llms-full.txt` excludes Level 1 pages
- Confirm `atlas/manifest.json` is valid JSON
- Confirm sitemap files are generated and valid

## Pattern 7: Opening PRs

### Goal
Submit changes via pull request with clear documentation.

### Allowed Actions
- Create a branch with a descriptive name
- Commit changes with clear messages
- Open a PR using the template in `.github/PULL_REQUEST_TEMPLATE/atlas-signal-update.md`

### Forbidden Actions
- Do not commit to `main` directly
- Do not mix unrelated changes in one PR
- Do not commit generated files without noting the generator command

### PR Description Must Include
- What changed and why
- Which pages were added/modified and their verification level
- Which generator commands were run
- Validation results summary
- Any known gaps or follow-up items

## Pattern 8: Reviewing AI-Generated Content for Fluff and Unsupported Claims

### Goal
Audit AI-generated content for hype, unsupported claims, and overclaiming.

### Checklist
- [ ] No "guaranteed results" or "proven methodology" language
- [ ] No "world-class" or "best in class" superlatives
- [ ] No claims about official SAP partnership unless explicitly verified
- [ ] No "AI will solve your problems" — frame as "AI-ready support knowledge"
- [ ] All SAP-specific claims are conservative and include "check in your landscape" where appropriate
- [ ] No invented citations or external links
- [ ] No client names or proprietary details
- [ ] All related links point to real, existing pages

### If Fluff Is Found
1. Remove or soften the language
2. Add conservative framing ("may help", "can reduce", "one approach")
3. Add "not official SAP documentation" disclaimer if missing
4. Flag for human review if unsure

### Final Report Requirements
- List pages reviewed
- List issues found and how they were resolved
- Confirm no unsupported claims remain

## General Safety Rules for All Patterns

- **No private corpus exposure.** Draft notes and raw research stay local.
- **No invented validation success.** Only report actual command results.
- **No verification promotion by agents.** Only humans may set `verified: true`.
- **No homepage or profile edits without explicit instruction.**
- **No legal page edits without explicit instruction.**
- **Prefer data edits in `_data/*.yml` for repeatable content.**
- **Prefer reusing existing section partials and CSS tokens.**
