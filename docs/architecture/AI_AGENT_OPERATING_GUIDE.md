# AI Agent Operating Guide — dkharlanau.github.io

Status: draft — internal documentation for maintainers and AI agents.  
Last updated: 2026-06-12

## How an AI agent should inspect the repo before work

1. Read `AGENTS.md` in the repo root.
2. Read `README.md`, `ARCHITECTURE.md`, and `PROJECT_MAP.md` for orientation.
3. Check the working tree:
   ```bash
   git status --short
   git branch --show-current
   git log --oneline -10
   ```
4. Note any dirty or untracked files. Do not touch files unrelated to your task.
5. Identify the content type you are changing using `docs/site-structure-inventory.md` §7.
6. Read the nearest existing page of the same type and copy its frontmatter/structure.
7. Confirm Ruby/Python availability if validation is required.

## Which files are safe to edit

- `_data/*.yml` — repeatable content (resume, home, certifications, changelog, etc.).
- `atlas/**/*.md` — new or existing Atlas pages (must start at Level 1 unless instructed otherwise).
- `scenarios/*.md` — new or existing Scenario pages (default to Level 1).
- `research/**/*.md` — research briefs, comparisons, watchlists, and synthesis files.
- `skill-hub/**/*.md` — Skill Hub pages and agent skill source files.
- `agent-skills/skills/**/*.md`, `agent-skills/skill-index.yml`, `agent-skills/profiles/*.yml` — agent skill sources.
- `datasets/**/*.json` — dataset entries (follow schema; run enrichment after edits).
- `ai/*.json`, `ai/*.yml` — machine-readable exports (when generated by scripts).
- `docs/**/*.md` — internal documentation (like this file).
- `_includes/atlas/*.html` — Atlas components.
- `_includes/sections/*.html` — section partials.
- `scripts/*.py`, `bin/*.py` — generators and validators.
- `tests/*.py` — tests.

## Which files require caution

- `index.md` — homepage. Edit only when explicitly instructed.
- `about.md` — canonical profile. Edit only when explicitly instructed.
- `services/*.md` — commercial service pages. Edit only when explicitly instructed.
- `legal/*.md` — legal policies. Edit only when explicitly instructed.
- `_config.yml` — affects the whole site. Changes can break builds or URLs.
- `robots.txt` — crawler policy. Edit only when explicitly instructed.
- `.github/workflows/*.yml` — CI. Edit only when explicitly instructed.
- `Gemfile`, `Gemfile.lock`, `.ruby-version` — dependency pins. Do not change without a strong reason.
- `_layouts/*.html` — affects every page.
- `_includes/head.html`, `_includes/seo/structured-data.html` — affects SEO and structured data globally.

## How to avoid publishing unverified content

- Always default new Atlas and Scenario pages to Level 1:
  ```yaml
  status: needs_verification
  verified: false
  robots: noindex,follow
  sitemap: false
  ```
- Never set `verified: true` without explicit human instruction.
- Never set `robots: index,follow` or `sitemap: true` on a page that has not been reviewed.
- If you promote a page, the human reviewer must update frontmatter; the agent must then regenerate artifacts.
- Do not add links to private or unverified content as if it were authoritative.

## Atlas noindex / sitemap / verified rules

The repository uses a three-level verification model:

| Level | Status | Frontmatter | Indexing |
|---|---|---|---|
| 1 — Review candidate | `needs_verification` | `verified: false`, `robots: noindex,follow`, `sitemap: false` | Noindex, excluded from sitemap and `llms-full.txt` |
| 2 — Verified | `reviewed` | `verified: true`, `robots: index,follow`, `sitemap: true` | Indexable, included in sitemap and `llms-full.txt` |
| 3 — Flagship | `reviewed` + source-backed | Level 2 plus citations, strong links, recent `last_reviewed` | Strong indexable signal |

Rules:
- Only Level 2+ pages appear in `llms-full.txt`.
- Only Level 2+ pages appear in section sitemaps.
- `scripts/generate_atlas_artifacts.py` enforces these rules when regenerating artifacts.
- `scripts/audit_discovery_outputs.py` checks the built site for violations.

## How to avoid duplicate pages

1. Search for existing pages with similar titles or topics before creating a new one.
2. Use `atlas/manifest.json` and `ai/atlas-compact-index.json` to find existing Atlas coverage.
3. Check `docs/site-structure-inventory.md` and `PROJECT_MAP.md` for existing routes.
4. Prefer updating an existing page over creating a near-duplicate.
5. Keep one canonical location per claim and link across pages instead of copying text.

## How to avoid private corpus leakage

- Never commit raw LinkedIn exports, ticket dumps, client names, incident IDs, or proprietary configuration.
- Never include local file paths (`/Users/...`, `.env`, `private-source`, `kb-drafts`) in public content or generated artifacts.
- Keep draft notes in local working directories or in noindex areas (`atlas/research-notes/`, `research/`).
- Generalize real examples: "a retail client" not "Client X"; "a repeated goods-receipt issue" not "Ticket 1234567".
- Run `python3 scripts/check_public_repo.py` before committing to catch disallowed paths and secret-like strings.

## Required validation commands

Run the smallest relevant set first. Run the full sequence before publishing.

```bash
# Python-only (can run without Ruby)
PYTHONDONTWRITEBYTECODE=1 python3 -m pytest tests
python3 scripts/check_public_repo.py
python3 scripts/generate_atlas_artifacts.py --check
python3 scripts/validate_site_content.py --strict

# Ruby/Jekyll required
bundle exec jekyll build
python3 scripts/check_links.py _site
python3 scripts/check_seo.py _site
python3 scripts/audit_discovery_outputs.py _site
```

If your environment lacks Ruby, run the Python-only checks and explicitly state that Jekyll build and post-build checks were deferred.

## Required final report format

After completing any task, report:

1. **Branch** and **commit SHA**.
2. **Files changed** with a one-line reason for each.
3. **Verification levels** of any new or modified Atlas/Scenario pages.
4. **Validation commands run** and exact results (pass/fail, output summary).
5. **Generator commands run** if artifacts were regenerated.
6. **Safety confirmations:**
   - No client names, ticket numbers, or proprietary details exposed.
   - No private paths leaked.
   - No generated files edited by hand.
   - No unverified content marked as verified.
7. **Known gaps** or follow-up items for human review.

## Common failure modes

| Failure | Likely cause | Fix |
|---|---|---|
| `generate_atlas_artifacts.py --check` fails | Artifact counts/content out of sync | Run `python3 scripts/generate_atlas_artifacts.py` |
| `check_links.py` reports broken links | Missing target page or typo in `href` | Verify permalink exists and fix link |
| `check_seo.py` fails | Missing title/description or canonical mismatch | Add frontmatter or fix absolute URL config |
| `check_public_repo.py` fails | Disallowed path or secret-like string in tracked files | Remove or sanitize the offending content |
| `pytest tests/test_atlas_artifacts.py` fails | Stale manifest or related.json | Regenerate artifacts |
| Jekyll build fails | Syntax error in Liquid/Markdown or missing dependency | Fix syntax or run `bin/setup` |
| `audit_discovery_outputs.py` fails | noindex page leaked into sitemap or private path in output | Fix frontmatter and rebuild |

## Stop conditions

Stop and ask the user before continuing if:

- The working tree has dirty files unrelated to your task and you are unsure whether to proceed.
- You are asked to edit `index.md`, `about.md`, `services/*.md`, `legal/*.md`, `Gemfile`, `Gemfile.lock`, `.ruby-version`, `_config.yml`, `robots.txt`, or `.github/workflows/*.yml` without explicit permission.
- You are asked to set `verified: true` or promote a page from Level 1 to Level 2.
- You are asked to delete files.
- A validation command fails and the fix is not obvious.
- You need to expose any information that might be private or proprietary.
- Ruby/Jekyll is unavailable and the task requires a full build validation.
