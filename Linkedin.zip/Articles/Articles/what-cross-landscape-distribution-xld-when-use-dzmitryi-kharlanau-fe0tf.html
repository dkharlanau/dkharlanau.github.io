What is Cross-Landscape Distribution (XLD) and when to use it?

Created on 2025-07-21 18:41

Published on 2025-07-21 18:54

You change config in one dev system. But QA is running on a different landscape. Pre-Prod too. Someone forgets a transport. A table isn’t updated. Replication fails. Now you’re debugging something that shouldn’t have broken in the first place.

XLD solves that.

It lets you take transports from one landscape and cleanly push them to another — across system groups, without copy-paste, without chaos.

This matters for SAP MDG, where you often have a central hub and multiple satellite systems. Config tables need to stay aligned — domains, key mapping logic, field checks, replication settings. If they drift, you’re done.

What You Can Actually Do with XLD

  • Copy transports from one Change Document into another in a different landscape

  • Or push directly into a target transport request

  • Use Strict Mode to define what’s allowed (specific packages, customizing tables)

  • Lock targets to prevent accidental changes

  • Overwrite objects even if someone already tweaked them in the target

Supports both manual (wizard-based) and automated modes — so you can control the flow or set it and forget it.

Is It Available Outside SolMan?

No — this is part of SAP Solution Manager 7.2+, specifically the Focused Build extension.

To use it, you’ll need:

  • Solution Manager properly set up with landscapes and logical components

  • Focused Build activated

  • RFC connections between SolMan and your SAP systems

  • Business role

  • Change Request Management (ChaRM) running

  • And optionally: strict rules on what’s allowed to be distributed

In MDG projects, configuration is just as critical as code. And config moves — value mappings, validations, key mapping behavior — all need to stay in sync across the entire landscape.
XLD gives you a repeatable, controlled way to move that config — no hacks, no side emails, no manual rework.
