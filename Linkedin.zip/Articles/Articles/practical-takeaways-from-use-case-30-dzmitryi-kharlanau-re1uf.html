I just finished reading Use-Case 3.0 by Ivar Jacobson and his team.

It’s not just another theory-heavy book; it’s packed with practical advice for anyone working with business requirements and systems. If you’ve ever been lost in endless documentation or vague goals, this book shows you how to cut through the noise and deliver real results.

Here’s the good stuff I learned and how we can use it.

What is Use-Case 3.0?

Use-Case 3.0 is a simple way to plan and build systems. It breaks big ideas into small steps you can build, test, and improve quickly.

Underlying Principles of Use-Case 3.0

  • Use cases aren’t just for software—they can describe any system: IT, business processes, hardware, or a mix of them all.

  • Use cases show the full purpose of a system and how it will be used.

  • Use cases revolve around users’ goals and how the system will help them.

  • Involve all key people early to define the system’s intent and scope.

  • A use case is like a complete narrative—from the trigger event to success or failure, including how errors are handled.

  • Talking about scenarios reveals missing steps, gaps, and overlooked alternatives.

  • Write so everyone understands. Clear communication beats complexity.

  • Begin with a basic sketch of events. Add more specifics when required.

  • Deliver small, valuable pieces first. Handle less critical parts later.

Core Concepts of Use-Case 3.0

The system (#1 A system of interest) you’re building or analyzing, whether it’s software, hardware, or a mix.

Think of it as: The “thing” that actors interact with to achieve their goals.

Anyone or anything (#2 An actor with a goal) that interacts with the system to achieve a goal. Actors can be people, other systems, or devices.

  • Primary Actor: Starts the interaction (e.g., a user logging in).

  • Supporting Actor: Assists the system during the interaction (e.g., a payment gateway).

What the actor wants (#3 A set of scenarios) to achieve using the system.

Example: A buyer’s goal is to purchase a product.

A complete sequence of steps (#4 Scenario) from the start of a use case to the end.

  • Basic Scenario: The “happy path”—everything works as planned.

  • Extensions: Alternative paths or error-handling scenarios.

All the ways of using the system to achieve a goal. This includes the basic scenario, extensions, and any variations.

Think of it as: A full story from start to finish.

A diagram (#6 Use-Case Model) showing all the use cases and their relationships to actors.

Why it’s useful: It gives a big-picture view of what the system will do.

A small, actionable part of a use case (#7 Use-Case Slice) that delivers clear value. It can be developed and tested on its own.

Example: For the use case “Withdraw Cash,” slices could be:

  • Basic withdrawal with a correct PIN.

  • Handling an incorrect PIN.

Let’s take an example of implementing a new sales order approval process in Sales. This ties directly into the concepts from Use-Case 3.0 and shows how they work .

1. System of Interest

The SAP SD module, where the sales order approval process will be implemented.

2. Actor

  • Primary Actor: Sales Manager who needs to approve or reject orders.

  • Supporting Actor: Customer Service Rep (CSR), who creates and submits sales orders for approval.

3. Goal

The goal of the process is to allow sales managers to approve high-value sales orders before they are processed.

4. Scenarios

Basic Scenario (Happy Path):

  1. CSR creates a sales order in SAP SD.

  2. The system identifies the order as requiring approval (e.g., value > $10,000).

  3. SAP sends the approval task to the Sales Manager.

  4. Sales Manager reviews the order and approves it.

  5. The system releases the order for further processing (e.g., delivery creation).

Extensions (Error Handling):

  • Extension 1: The Sales Manager rejects the order → SAP sends rejection notification to the CSR → CSR updates or cancels the order.

  • Extension 2: The Sales Manager doesn’t respond within 24 hours → The system escalates the task to the Regional Manager.

  • Extension 3: The order has invalid data → SAP notifies the CSR to correct it before resubmitting.

5. Use Case

The Sales Order Approval use case includes:

  • Basic approval process.

  • Handling rejections, escalation, and invalid data scenarios.

6. Use-Case Model

This can be visualized with a diagram:

Actors:

  • Sales Manager

  • CSR

  • Regional Manager

Use Cases:

  • Approve Sales Order

  • Reject Sales Order

  • Escalate Pending Approval

7. Use-Case Slices

You don’t build the entire process in one go. Break it into slices:

  • Slice 1: Sales order creation and triggering approval workflow.

  • Slice 2: Approve or reject the order.

  • Slice 3: Handle escalation for delayed approvals.

  • Slice 4: Process notifications for invalid orders.

Each slice is testable and delivers incremental value, ensuring early feedback and faster delivery.
