BRFplus: Make Business Rules Work for You

Created on 2024-12-04 15:49

Published on 2024-12-04 19:15

Business rules are everywhere—whether it’s deciding who gets a discount, setting approval workflows, or managing tax calculations. But let’s be real: rules buried in code or scattered across handbooks slow everything down. That’s where SAP BRFplus comes in. It’s a tool designed to centralize, simplify, and supercharge how you manage business rules.

Here’s the deal: BRFplus lets you define and control your business rules in one place, decoupled from the main application logic. That means you can tweak rules on the fly without needing a full system update. I’ve used it in several projects, and it’s a game-changer. Let’s break it down.

Why BRFplus?

  1. Centralized Control: Manage all your rules in one repository.

  2. Empower Business Users: Business experts define rules; IT supports implementation.

  3. Decoupled Logic: Rules are separate from application code, making updates fast and risk-free.

  4. Flexibility: Update rules in real-time to adapt to changing business needs.

Getting Started with BRFplus

Prerequisites

  • Authorization: Assign the role to users via transaction .

  • Naming Conventions: Follow these standards for clarity and maintainability

For interfaces and classes, BRFplus uses the following conventions:

  • Interfaces start with prefix IF_FDT_...

  • Classes start with prefix CL_FDT_...

  • Exception classes start with prefix CX_FDT_...

Within these objects the following naming conventions apply for attributes:

  • Constant definitions start with prefix GC_...

  • Static attribute definitions start with prefix GV_...

  • Member attribute definitions start with prefix MV_...

Parameter names within class and interface methods (or also function modules) have a prefix that consists of two to three characters. The first character defines the parameter’s usage type:

  • I for importing parameters

  • E for exporting parameters

  • C for changing parameters

  • R for returning parameters

The second character of the prefix describes the parameter’s data type:

  • V for simple variable types

  • S for structure types

  • T for table types

  • R for reference types

  • O for object types

  • A for generic/unspecific data types like ANY or DATA

For table types an additional third character may further refine the type of the

parameter:

  • H for hashed table types

  • S for sorted table types

Core Concepts

Applications

  • Logical units for grouping BRFplus artifacts, similar to a project folder.

  • Use namespaces (, ) to organize customer-specific objects.

  • Versioning allows tracking changes and running older versions of objects when needed.

Storage types:

  • System Objects: Cross-client, transported via Workbench requests

  • .Customizing Objects: Client-dependent, transported via Customizing requests.

  • Master Data Objects: Client-specific, not transportable.

Functions

Define the entry point for rules processing.

Two modes of operation:

  • Functional Mode: Executes a single expression and returns the result.

  • Event Mode: Triggers one or more rulesets for complex processing.

Rulesets
  • Encapsulate rules and define conditions for their execution.
  • Subscribe rulesets to functions in event mode for dynamic execution.
  • Set priorities for ruleset execution order.
Rules
  • Implement IF-THEN-ELSE logic.
  • Can include operations like:Resetting context data.Setting or exchanging values.Executing nested rules or external actions.
Expressions: Building Blocks of Logic
Expressions are reusable components that define specific logic in BRFplus. Key types include:
  1. Decision Table
  2. Boolean Logic
  3. DB Lookup
  4. Formula
  5. Case
  6. Decision Tree
  7. Procedure Call
Actions: Triggering External Activities
Actions go beyond BRFplus, performing tasks like:
  • Log Message: Adds entries to application logs ().
  • Send Email: Notifies stakeholders.
  • Start Workflow: Triggers SAP workflows.
  • Procedure Call: Calls external services via APIs.
Integration with Other Frameworks
BOPF Integration
  • Use the /BOFU/RETRIEVE_BO_DATA expression to access Business Object data in BRFplus.
  • Example: Retrieve customer or sales order data for rules processing.

When to Use BRFplus
BRFplus is ideal when:
  • You need to centralize business logic across multiple systems.
  • Rules are complex or frequently updated.
  • Business users should control rule maintenance without involving developers.
What I Learned
BRFplus shines when you need flexibility and speed. On one project, we used it to streamline tax calculations across regions. The rules changed often, but with BRFplus, business users could update them instantly without breaking the system.
