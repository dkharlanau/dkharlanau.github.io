HTTP vs WebSockets vs gRPC ‚Äî What You Need to Know
üåê HTTP(S) ‚Äî Classic Request/Response
  • Format: JSON / XML
  • One request ‚Üí one response
  • Use: SAP OData, REST APIs
  • ‚úÖ Fully supported in Postman
Standard for CRUD, SAP integration, backend calls.
üîÅ WebSockets (WSS) ‚Äî Real-Time Channel
  • Format: Text / Binary
  • Persistent, bidirectional
  • Use: Chat, live updates, event streams
  • ‚úÖ Supported in Postman (wss:// URLs)
Great for testing push messages and subscriptions.
‚ö° gRPC ‚Äî High-Speed Binary RPC
  • Format: ProtoBuf (binary over HTTP/2)
  • Faster, strongly typed
  • Use: Microservices, backend-to-backend
  • ‚úÖ Supported in Postman (with  import)
Best for low-latency, high-volume systems.

Pre-request & Test Scripts in Postman
Postman isn't just a UI for sending requests ‚Äî it's a mini runtime. You can write JavaScript to control, prepare, and validate your calls.
1. Pre-request Script ‚Äî Run Before the Request
Used to:
  • Set or modify variables
  • Generate timestamps, auth tokens, dynamic IDs
  • Send chained calls (e.g., get token ‚Üí use it in main request)
2. Test Script ‚Äî Run After the Response
Used to:
  • Validate response code, body, headers
  • Store values (e.g., order ID) for the next request
  • Write custom assertions
