Building Multitenant Applications on SAP BTP

Created on 2025-05-19 09:42

Published on 2025-05-19 20:15

When you're building apps for business, sooner or later you’ll hear this:

“Can other customers use this app too?”

If your answer is “Yes, but we have to install it again for each one” — that’s a sign you need to think about multitenancy.

❓What Is a Multitenant App?

A multitenant application is one app that can serve many customers (tenants) at the same time. Each customer:

  • Uses the same code

  • Runs on the same infrastructure

  • Has separate data, users, settings

You deploy once, but can scale to hundreds of tenants.

🎯 Why Multitenancy?

✅ Easier to manage

✅ Lower costs (one system, many clients)

✅ Fast updates for all tenants

✅ Strong isolation and security

For SAP BTP, this is the recommended way to build SaaS solutions.

🧱 How It Works on SAP BTP

Here’s how to design a multitenant app on SAP BTP (Cloud Foundry):

1. Provider Account

This is where you develop and deploy your app. Here you register your app as “SaaS-enabled” using:

  • SaaS Provisioning Service — handles subscription/unsubscription

  • Application Router — routes users to the backend

  • XSUAA — manages login and tells your app which tenant is calling

  • Multitenant CAP/RAP backend — knows how to react based on tenant ID

  • Database strategy — one schema per tenant (HDI containers) or shared with strict filters

2. Subscriber Accounts (Tenants)

Each customer has their own subaccount in SAP BTP. They subscribe to your app. The platform automatically handles:

  • Role and access setup

  • OAuth token exchange

  • Routing to your backend

  • Isolating the tenant context

🔐 Key Design Rules

  • Never trust frontend: always use the JWT token (XSUAA) to get the tenant ID

  • Always isolate data: separate schema or strict tenant ID filters

  • Use tenant-aware services: destinations, identity, etc.

  • Build with onboarding in mind: prepare callback handlers to configure tenant when they subscribe

💰 How to Estimate Costs

On SAP BTP, your cost depends on:

Each tenant does not pay individually — you pay once for the shared setup. You can charge tenants based on data size, users, or features.

🗂️ Summary for Teams & Customers

You can explain it like this:

“We will build the app once. Each customer gets access with their own login and data. They won’t see each other. We can scale to 10, 100, or 1000 customers without copying the app. And we control everything — updates, security, and performance — from one place.”

It’s faster, cheaper, and easier to maintain than having 10 separate apps.
