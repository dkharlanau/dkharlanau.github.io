Why Extensions Are the Last Resort for SAP S/4HANA Cloud (and How to Do It Right)

Created on 2024-11-06 13:53

Published on 2024-11-06 20:30

In the SAP world, everyone’s talking about the clean core—keeping your system as close to the original, out-of-the-box state as possible. Why? Because every extension you add makes things more complex, harder to maintain, and potentially a nightmare during upgrades. So, let’s dive into why extensions should be avoided (whenever you can) and the right way to approach them when you can’t.

Keep it Simple: Don’t Add Extensions if You Don’t Have To

The first rule of the clean core? Don’t add extensions unless absolutely necessary. Before you start developing something new, take a good look at what SAP S/4HANA Cloud already offers. Many of the features you need might already be there, ready to use, just waiting for a bit of configuration. Only when your business processes have been optimized and still have gaps that standard SAP can’t fill should you even think about extensions. Keeping extensions to a minimum keeps your ERP lean, reducing complexity and future headaches.

If You Must Extend, Keep It Structured

So, you’ve decided an extension is essential. Here’s the deal: SAP requires a solid governance structure to approve extensions, meaning every extension needs a clear purpose and must add significant value. Companies should document why the extension is necessary and how it will enhance business functions, rather than just adding it because it seems convenient.

Extensions vs. Upgrades: Why Compatibility Matters

An extension that isn’t “clean core-compliant” can break your ERP system during upgrades, causing anything from minor glitches to full-scale failures. To avoid this, SAP recommends creating extensions that adapt seamlessly during system upgrades. The trick? Ensuring the extension is separate from SAP S/4HANA Cloud code, built using SAP’s extensibility model. This way, when SAP rolls out updates, your extensions don’t suddenly collapse.

Let’s break it down by extension type:

  1. On-stack extensions (within SAP S/4HANA Cloud): Key-user extensions or developer extensions that need close integration with the core.

  2. Side-by-side extensions (on SAP Business Technology Platform): Great for features that don’t need to sit directly in the ERP, keeping things modular.

Using this approach gives you a clear boundary between SAP’s code and your customizations.

Stick to “Released” Tools Only

Another key clean-core rule: only use approved APIs and extension points like BAdIs or BO extension points. This keeps things transparent, so everyone knows what’s custom versus what’s SAP. Non-released objects are tempting but risky since they don’t have SAP’s guarantee for compatibility, which could mean trouble down the line.

Cloud-First Extensions: ABAP Cloud and the Three-Tier Model

For extensions to work smoothly in the cloud, SAP introduced some guidelines. Here’s the quick version:

  1. ABAP Cloud Development Model: This is SAP’s way of adapting ABAP for the cloud—keeping the language familiar but with some restrictions. Only approved ABAP Cloud objects are allowed, and syntax checks ensure that only compatible code makes it in.
