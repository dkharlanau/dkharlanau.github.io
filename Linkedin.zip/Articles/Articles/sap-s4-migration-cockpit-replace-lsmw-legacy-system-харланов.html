Will SAP S4 Migration Cockpit replace LSMW - Legacy System Migration Workbench?

Created on 2021-12-27 17:44

Published on 2021-12-27 18:35

With updates or with the shift to the new technology IT companies constants try to develop the tools to make them faster, more reliable, more convenient for use. With the shift to SAP S4 from ERP ECC SAP has provided a Migration Cockpit to manage and upload master data for migration purposes and make it modern. Before this, the main tool for creating master/transactional data or automation of manual process in SAP ERP was LSMW - Legacy System Migration Workbench. Many SAP consultants and ABAP developers remember the 10+ footway. Do you remember it?

Can we forget about LSMW - Legacy System Migration Workbench after upskilling to Migration Cockpit?

To answer this question, we should have a quick look at the use of Migration Cockpit. It is anticipated, new transactions run in SAP S4 using Fiori what should simplify the user experience and when transactions should be role and process-oriented. The Migration Cockpit is not the exclusion and it will guide a user through necessary steps.

Select a way to upload/migrate the data:

  1. Transferring Data to SAP S/4HANA Using Files. The SAP S/4HANA migration cockpit uses Microsoft Excel XML files. 

  2. Transferring Data to SAP S/4HANA Using Staging Tables: The SAP S/4HANA migration cockpit can automatically create staging tables for each migration object that is relevant for your project. You can fill the staging tables with data either manually or by using your preferred tools (for example SAP Agile Data Preparation).

  3. Migrate Data Directly from SAP System.

Migration Cockpit is in action

We will use the option for transferring Data to SAP S/4HANA Using Files.

1. Create a migration  object ( up to S4 2020 it is the transaction LTMC )

2.  Select a migration objects that are relevant for the migration scenario 

Customer - extend existing record by new org levels - SAP Help Portal

Read all comments for the migration object available in documentation. In some objects there are restrictions that will not allow to run a migration scenario. Example with extending customers..

A list of available objects for migration: Available Migration Objects - SAP Help Portal
3. Download and fill template: each page of XML file ( can be opened with Excel ) will represent a view of business object
4. Process any mapping tasks for the migration objects: 
  • Every file should be activated and passed for transferring
  • Then we will be able to convert row value into the target value in S4 and check if the value can be mapped. A field should have the conversion rule in the migration object ( see later ).
5.  Simulate the migration before migrating your data to SAP S/4HANA
  •  if there are any errors, they are displayed 
 6. Migrate your data to SAP S/4HANA and monitor the status of the migration.
The process with the standard migration object has gone and still the questions? Thus, we should jump into SAP S/4HANA Migration Object Modeler (Transaction LTMOM) as it will explain what happens in the black box during migration run. 
SAP S/4HANA Migration Object Modeler
The migration object consists of several base elements:
  1. Global data, which includes the data and type definition 
  2. Source structure where we can maintain a list of linked sources. The header source (S_CUST_GEN)  contains a customer and sub source-node Sales Area ( S_CUST_SALES_DATA )
The fields can be freely set. 
3. Target structure. Here we should have a function module which will be called, the Modeler will pick input parameters of the FM automatically. It is the heart of the processing part.
4. Structure mapping: assign a node of the source structure to a an input of the function module, set the relationship for the linkage
5. Field mapping: each field can be now with Drag & Drop mechanism mapped to the target field of the function' s input. Not only from the source structure, but also with variables determined or calculated with the customer-specific rules.
6. Rules are containers with useful coding to retrieve values, execute pre-checks..
When comparing Migration cockpit and Legacy System Migration, it is evident:
Use LSMW when we have no API as functions to run updates or create messages for inbound processing. It is the case where the Transaction recorder is the best-fit choice
Use Migration cockpit in other cases. 
  1. It provides a user friendly way to struct and process the data. 
  2. The template file can contain a list of fields for uploading per the business object view, grouped and with meaningful comments. 
  3. It is already the list of standard objects that can be used out of the box to manage the data in SAP S4. 
  4. SAP will continue to extend and develop this tool in the next releases. 
  5. It is the great Modeler to adopt and create own objects for data upload.
Export / Import Project (Migrate Data Using Staging Tables)
With SAP S/4HANA 2020 release, you can transfer a project from one system to another in the migration object modeler (transaction LTMOM) using the export/import Project functionality.  
Use the report /LTB/MC_PROJ_Q2P to export and import.  
1. Migrate your data in JD1: copy the project, remember the new transfer ID ( like MOF )
2. Use the program /LTB/MC_PROJ_Q2P to export from JD1 and import to JP1
3. Use T-code LTMOM to activate ALL objects we have under the transfer ID
