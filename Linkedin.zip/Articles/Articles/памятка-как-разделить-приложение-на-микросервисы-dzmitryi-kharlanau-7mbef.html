Памятка: Как разделить приложение на микросервисы

Created on 2024-12-11 15:44

Published on 2024-12-11 20:30

Разделение приложения на микросервисы — это всегда баланс между техникой, бизнес-логикой и здравым смыслом. Неправильный подход может создать «распределенный монолит», который хуже монолита. Как этого избежать? Вот ключевые шаблоны и практики.

1. Паттерны декомпозиции

1.1 Разбиение по бизнес-возможностям

Суть: выделяем основные бизнес-функции (то, что видят и используют пользователи) и создаем для каждой из них отдельный микросервис.

Примеры:

  • Управление заказами

  • Управление клиентами

  • Управление оплатами

Эти микросервисы работают независимо, что упрощает масштабирование, обновления и развитие системы.

1.2 Разбиение по поддоменам (DDD)

Суть: используем Domain-Driven Design (DDD) и делим предметную область на поддомены.

  • Каждый поддомен имеет свою модель данных и область действия (Bounded Context).

  • Пример: В логистике можно выделить поддомены: управление поставками, управление складами и управление транспортировкой.

  • Главное: четко определить границы между поддоменами, чтобы микросервисы оставались автономными.

2. Как перейти к микросервисам из монолита?

Если у вас уже есть монолитное приложение, переход на микросервисы требует поэтапного подхода. Здесь помогут шаблоны рефакторинга.

2.1 Шаблон «Душитель» (Strangler)

  • Постепенно вытесняем функциональность монолита новыми микросервисами.

  • Новый микросервис замещает часть логики, монолит со временем «усыхает».

  • Плюсы: минимизация рисков и плавный переход.

2.2 Шаблон «Уровень защиты от повреждений» (Anti-Corruption Layer)

Создаем промежуточный слой, который:

  • Преобразует запросы и данные между новой и старой системой.

  • Изолирует микросервисы от устаревшего кода.

Пример: в SAP это может быть модуль, который «переводит» данные между старой логикой заказов и новой архитектурой.

3. Управление данными в микросервисной архитектуре

Разделение данных между микросервисами — это отдельный вызов. Вот основные шаблоны:

3.1 «База данных на сервис» (Database Per Service)

Каждый микросервис имеет свое хранилище данных.

Плюсы:

  • Максимальная автономность сервисов.

  • Меньше зависимостей между командами.

Пример: Сервис клиентов работает только с таблицей , а сервис товаров — только с таблицей , даже если обе таблицы находятся в одной базе.

3.2 «Разделяемая база данных» (Shared Database)

Общая база для всех микросервисов.

  • Минусы: сильная связанность сервисов.

  • Используйте только на старте или для небольших проектов.

3.3 «API-композиция» (API Composition)

Когда данные нужны из разных микросервисов, создаем API, который:

  • Обращается к нужным сервисам.

  • Собирает и объединяет данные для клиента.

  • Пример: единый API для фронтенда, который берет данные из микросервисов логистики, склада и платежей.

3.4 «Разделение команд и запросов» (CQRS)

  • Command: операции изменения данных.

  • Query: операции чтения данных.

  • Пример: для быстрого чтения используем реплики баз данных, а для записи — основной кластер.

3.5 «Распределенные транзакции» (Saga)

Управление транзакциями между микросервисами с помощью событий.

  • Оркестровка: централизованный компонент управляет всеми шагами.

  • Хореография: каждый сервис реагирует на события других.

4. Что помнить при проектировании?

  • Границы важны: чем лучше вы определите поддомены или бизнес-функции, тем автономнее будут микросервисы.

  • Избегайте сильной связанности: автономность — ключ к масштабируемости.

  • Не торопитесь: переход на микросервисы — это процесс, а не цель.
