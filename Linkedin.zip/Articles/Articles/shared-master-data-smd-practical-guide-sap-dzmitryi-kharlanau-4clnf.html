Managing master data changes across decentralized systems can feel like juggling flaming torches. Enter Shared Master Data (SMD)—SAP’s tool for distributing these changes effectively. Whether it’s materials, vendors, or customers, SMD ensures everyone’s playing from the same data sheet. Here’s how it works and how you can make it your best friend in maintaining master data consistency.

How Does SMD Work?

SMD integrates with the Change Document Interface ( and ) to track changes made to master data. When a change is made, it writes a change document that triggers the creation of change pointers. These pointers then drive the distribution process, sending updates via IDocs to the relevant systems.

Configuring Shared Master Data

  1. Activate Change Pointers Globally: Use transaction to enable change pointer functionality across the system.

  2. Activate Change Pointers for Specific Message Types: Enable change pointers for a particular object (e.g., materials, customers) using transaction .

  3. Filter Change Pointers: Fine-tune which fields trigger change pointers with transaction . This keeps unnecessary triggers from clogging the system.

  4. Reduce Change Pointer Data: Use transaction and refer to SAP Note 0420562 to minimize the number of change pointers created.

  5. Generate and Process IDocs: Create IDocs from change pointers with transaction or report . This ensures the updates are sent to all target systems.

  6. Clean Up Old Change Pointers: Regularly run transaction or report to delete processed and obsolete change pointers.

  7. Apply Custom Filters with a BAdI: Implement to filter out irrelevant changes before they create change pointers.
